## 🔴 CRITICAL — open (money-loss, data-leak, or DB DoS)

### D2 — reorderService N+1 query in hot loop
**File:** `rezbackend/rez-backend-master/src/services/reorderService.ts:111`
**What:** `for (const orderItem of itemsToReorder)` fires a `Product.findById().populate().lean()` per item, serially. A cart with 50+ items → 50+ serial round-trips per reorder click. During campaigns this pins the DB connection pool.
**Fix:** Batch with `Product.find({ _id: { $in: productIds } }).populate(...).lean()`, then map.
**Effort:** ~30 min

### D3 — Mongoose raw error messages returned to client (PII leak)
**Files:** `rezbackend/rez-backend-master/src/controllers/consultationController.ts:125,179,225,304,365,479`; `referralTierController.ts:50-628`; likely widespread
**What:** `res.status(500).json({ message: error.message })` with no filtering. Duplicate-key errors leak schema + PII: `E11000 duplicate key: { phoneNumber: "+91xxxxx" }`. CastErrors leak internal field names.
**Fix:** Central error-mapping middleware: map `ValidationError`, `MongoServerError:11000`, `CastError` to sanitized messages. Only pass through `message` for custom `AppError` subclasses you instantiated.
**Effort:** ~2-3h (touches many controllers; central middleware is the right fix)

## 🟠 HIGH — open

### D4 — 6 node-cron jobs have no distributed lock
**Files:**
- `rezbackend/rez-backend-master/src/jobs/rebookingNudgeJob.ts:198`
- `jobs/lifecycleAutomationJob.ts:234`
- `jobs/fraudDetection.ts:118`
- `jobs/archiveJob.ts:191`
- `jobs/priveInviteExpiryJob.ts:49`
- `jobs/automatedCampaignJob.ts:440`

**What:** In-memory `isRunning` guard only. Multi-replica deploy = every replica fires at the scheduled minute → double-sends, double-archives, double-nudge-pushes to users.
**Fix:** Wrap each with `redisService.acquireLock(key, ttl)` like `partnerLevelMaintenanceService.ts` already does; OR migrate to `ScheduledJobService` (BullMQ repeatable jobs, auto-serialized).
**Effort:** ~2h for the 6 jobs

### D5 — "Today" boundary uses server UTC, business logic is IST
**Files:**
- `rezbackend/rez-backend-master/src/services/ScheduledJobService.ts:431,325`
- `services/leaderboardService.ts:660-710`
- `services/cashbackService.ts:472-493`

**What:** `new Date().setHours(0,0,0,0)` computes midnight in server-local (UTC on Render). Users in IST can double-dip daily cashback caps by transacting in the 18:30–23:59 IST window (5.5h between IST midnight and UTC midnight). Leaderboard "daily" also skews.
**Fix:** Shared `startOfDayIST(date)` helper using `Asia/Kolkata` timezone. Normalize all day-boundary financial aggregates.
**Effort:** ~1-2h (well-bounded, high-leverage)

### D6 — Razorpay calls lack timeout + socket-leak on "timeout"
**File:** `rezbackend/rez-backend-master/src/services/razorpayService.ts:69-76,148-156,200-208`; also `routes/userSubscriptionRoutes.ts:188,208`
**What:** `razorpayService` uses `Promise.race([sdkCall, timeout])` — the timeout resolves but the underlying SDK socket stays open until TCP timeout (~120s), leaking sockets under load. `userSubscriptionRoutes` calls Razorpay plan/subscription endpoints with NO timeout at all.
**Fix:** Use the Razorpay SDK's native `timeout` option (on the constructor) or wrap the HTTP layer with a real AbortController. Already have `withGatewayTimeout` pattern in `paymentGatewayService` — extend to these sites.
**Effort:** ~1h

### D7 — SendGrid has no timeout / circuit breaker
**File:** `rezbackend/rez-backend-master/src/services/EmailService.ts:61`
**What:** `sgMail.send()` with no wrapper. SendGrid outage → every request handler waiting on the ~2-minute default TCP timeout. No `withCircuit` like `SMSService` uses for Twilio.
**Fix:** Mirror the `twilioCircuit` + `withTwilioTimeout` pattern for SendGrid.
**Effort:** ~30 min

### D8 — BullMQ `removeOnComplete: { age: N }` has no count cap
**Files:**
- `rez-payment-service/src/worker.ts:197`
- `rez-order-service/src/worker.ts:311`
- `rez-gamification-service/src/worker.ts:371`
- `rez-gamification-service/src/worker.ts:372` (removeOnFail)
- `rez-notification-events/src/worker.ts:614`

**What:** 100k payment completions in the 1-hour `age` window keeps all 100k job payloads in Redis. Each contains `{ userId, amount, idempotencyKey }`. Redis memory pressure → eviction → real cache keys evicted.
**Fix:** Add `count: 10000` alongside `age` on each definition.
**Effort:** ~15 min

### D9 — Expiring-coins notification cron does multikey index scan on 500k+ wallets
**File:** `rezbackend/rez-backend-master/src/services/ScheduledJobService.ts:118-130`
**What:** Only index on `coins` is `{ 'coins.expiryDate': 1, sparse }`. Daily job does `Wallet.find({ coins: { $elemMatch: { type, isActive, expiryDate } } })` — multikey scan at 500k+ wallets takes 30s+ daily.
**Fix:** Add compound multikey index `{ 'coins.type': 1, 'coins.isActive': 1, 'coins.expiryDate': 1 }`.
**Effort:** ~15 min (migration)

## Pattern observations

1. **Date/TZ is the most pervasive hidden bug** — 15+ places compute "today" with `setHours(0,0,0,0)` against server UTC while business rules are IST. No shared `startOfDayIST` helper exists. Highest-leverage single refactor in this list.

2. **`removeOnComplete: { age: N }` without a count cap** — pattern repeated in 5+ workers. Silent Redis memory-growth risk under burst.

3. **Third-party timeout handling is split-brain** — `paymentGatewayService` uses proper wrappers, `SMSService` uses circuit breakers, but `razorpayService` uses Promise.race (socket leak), `EmailService` has no wrapper, `userSubscriptionRoutes` has nothing at all.

4. **Cron distributed locking is inconsistent** — `ScheduledJobService` (BullMQ-based) auto-serializes, but at least 6 `node-cron` jobs have only in-memory guards.

5. **N+1 populate-in-loop is a widespread pattern** — 16+ services on initial grep. Worst offenders: `reorderService`, `specialProgramService` Privé recheck, `referralFraudDetection` on the write path.

## Launch-blocker status

**None of D2–D21 are launch-blockers in the same sense as B1–B12 were.** They're quality-and-scale issues that matter most:
- Under load (D2, D9, D10)
- After 4-6 hours of live traffic (D6 socket leaks, D7 SendGrid blocked handlers, D8 Redis memory)
- On a DST / midnight-IST transition (D5)
- When a user triggers a refund or error that surfaces a Mongo schema (D3)
- On a 2nd replica deploy (D4)

If you're launching with 1 replica + small initial traffic, most of these can wait a week or two post-launch. **D1 (ReDoS) was truly launch-blocking and is now fixed.**
